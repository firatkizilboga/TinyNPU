\section*{\centering\thetitle}
\centerline{\fontsize{16pt}{21.6pt}\sffamily\bfseries (SUMMARY)}
TinyNPU is a compact accelerator implemented as a complete hardware/software stack. It combines a parameterized systolic-array neural processing unit with a PyTorch-first segmented compiler and runtime, a quantization-aware model-preparation flow, RTL and host-side verification infrastructure, and performance and synthesis characterization tooling. The stack is designed to make the datapath usable from a realistic machine-learning workflow and measurable in a disciplined way.

The hardware architecture is organized around an $N \times N$ output-stationary systolic array, a unified buffer, an instruction memory, streamer logic that feeds the array, and a post-processing unit that applies bias, rescaling, activation, and packed writeback. The software stack traces PyTorch models, partitions them into accelerator segments and host operations, compiles supported segments into TinyNPU instruction and buffer images, and executes the resulting plan on either host emulation or RTL simulation. Unsupported work is never hidden; it appears explicitly as host-side steps in the execution plan.

The useful unit of design is the whole stack, not the RTL alone. Compiler-visible tensor semantics, physical packing format, host/NPU boundaries, verification points, and repeated-run runtime behavior all affect whether the accelerator is practical to use. TinyNPU therefore includes explicit host-op dispatch, compiler-owned expected tensors for verification, repeated-run sessions with persistent state, and benchmark accounting that separates accelerator compute cycles from runtime and protocol overhead.

The implemented stack has been validated end-to-end on a trained MNIST model, and a controlled multi-tile matrix multiplication benchmark is used to study compute speedup and integration overhead. In addition, a reproducible Yosys-based synthesis flow provides rough resource estimates for the current design. The report presents TinyNPU as a working accelerator research system for studying architecture, compilation, packing, and workload fit together.

\section{Introduction and Project Summary}
\subsection{Background and Motivation}
Modern deep neural networks demand substantial compute and memory bandwidth even at inference time. This has made specialized accelerator design a central topic both for datacenter-scale systems and for smaller edge-oriented deployments \cite{jouppi2017datacenter, genc2021gemmini}. Quantization is one of the main techniques that makes such deployment practical, because it reduces memory traffic and allows arithmetic throughput to increase when narrow integer types are used instead of floating-point data.

However, quantization is not a single fixed target. Mixed-precision work has shown that different layers or kernels inside the same model can tolerate different bit-widths \cite{wu2018mixedprecision}. This creates a hardware opportunity and a software problem at the same time. Hardware should benefit from narrower arithmetic when it is safe to do so, but the software stack must still prepare the model, preserve numerical intent, and decide where accelerator execution ends and host execution begins.

\subsection{Problem Definition}
The main problem addressed by TinyNPU is therefore not only how to build a small systolic-array accelerator, but how to make mixed-precision quantized computation usable from a realistic model flow. Fixed-precision accelerators leave throughput on the table when different parts of a model have different precision needs. At the other end of the spectrum, CPU-centric approaches based on ISA extensions or soft SIMD can support low-bit computation, but they still pay substantial software-side cost for packing, unpacking, and orchestrating the data movement around those operations \cite{garofalo2021xpulpnn, armeniakos2024mixedprecision, reggiani2023mixgemm}.

This project treats the problem as a co-design task. The hardware must expose a regular datapath that benefits from packed low-precision arithmetic. The compiler and runtime must make tensor layout, quantization boundaries, and host fallbacks explicit rather than implicit. The result should be a system that can be measured honestly: accelerator compute, runtime overhead, and host-side work should not be conflated.

\subsection{Project Scope and Approach}
TinyNPU approaches this problem with four coupled components:
\begin{enumerate}
    \item \textbf{A parameterized mixed-precision accelerator:} TinyNPU is centered around an output-stationary $N \times N$ systolic array, a unified buffer, streamer logic that feeds the array, and a post-processing unit that performs bias addition, rescaling, activation, saturation, and packed writeback.
    \item \textbf{Packed integer computation:} The processing elements support packed INT16, INT8, and INT4 execution modes. Throughput therefore scales with precision reduction, not by changing the global architecture, but by packing more useful work into the same datapath.
    \item \textbf{A segmented compiler/runtime:} The software flow traces supported PyTorch models, partitions them into accelerator segments and explicit host-side operations, lowers legal accelerator work into instruction and buffer images, and executes the resulting plan on either host emulation or RTL simulation.
    \item \textbf{Verification and measurement infrastructure:} The project includes compiler-owned expected tensors, runtime verification points, repeated-run sessions, and benchmark accounting that separates accelerator compute from protocol and software overhead.
\end{enumerate}

TinyNPU runs a trained MNIST model through the full compiler/runtime stack and benchmarks a controlled multi-tile matrix multiplication workload. These experiments expose the architecture, packing behavior, compiler boundaries, and runtime cost of the current system.

The rest of this report follows that logic. The next section positions TinyNPU relative to relevant literature in mixed-precision quantization, matrix multiplication, and accelerator design. The system model section then explains the implemented hardware and software stack in detail, followed by the experimental method, the current results, and the most important next steps.

\section{Comparative Literature Survey}
\subsection{Mixed-Precision Quantization}
Mixed-precision quantization work is the clearest algorithmic motivation for TinyNPU. Wu et al. formulate layer-wise quantization as a search problem and show that different layers can be assigned different bit-widths without treating the whole model uniformly \cite{wu2018mixedprecision}. This is the key observation behind a variable-precision accelerator: if model layers do not all need the same precision, a fixed-precision datapath cannot exploit the full efficiency opportunity.

Quantization is also relevant as a compiler concern rather than only a hardware concern. SmoothQuant is an example of pushing difficult numerical adjustments into an offline transformation so that runtime execution becomes simpler and more hardware-friendly \cite{xiao2023smoothquant}. TinyNPU follows the same general philosophy even though it does not implement the full SmoothQuant method itself: model preparation, fused rescaling, and compiler-ready conversion are treated as software responsibilities that simplify the accelerator contract.

\subsection{High-Performance Matrix Multiplication}
The mathematical core of TinyNPU is still matrix-style computation, so the project is strongly informed by classical GEMM literature. Goto and van de Geijn show that high-performance matrix multiplication depends not only on arithmetic throughput, but on careful tiling and data movement through the memory hierarchy \cite{goto2008anatomy}. The same principle appears in TinyNPU at a smaller scale. When matrices exceed the dimensions of the systolic array, the compiler and runtime must tile along the $M$, $N$, and $K$ dimensions while preserving the data reuse pattern expected by the array and its post-processing path.

\subsection{Existing Accelerator Architectures}
On the accelerator side, Google's TPU remains the canonical reference for systolic-array-style matrix acceleration \cite{jouppi2017datacenter}. Gemmini is especially relevant because it demonstrates how a systolic array, programming model, and system context can be treated as one full-stack research object rather than as an isolated RTL block \cite{genc2021gemmini}. That full-stack perspective is one of the strongest methodological reference points for TinyNPU.

An educational TinyTPU implementation was a useful starting point for understanding systolic-array organization, but it was not a good endpoint for this project \cite{chin2025tinytpu}. Extending it toward mixed precision, tiled execution, and cleaner post-processing exposed the limits of treating the datapath as the whole problem. TinyNPU therefore diverged into a custom implementation with a different software contract, explicit host/accelerator boundaries, and a compiler/runtime that owns packing and verification rather than leaving those concerns informal.

Taken together, these references define the design space in which TinyNPU sits. It is not trying to outscale large industrial accelerators. Its niche is a compact mixed-precision accelerator system in which the hardware, packing model, compiler/runtime, and measurement methodology can all be studied together in one working platform.

\section{Developed Approach and System Model}
The system is described here as four cooperating subsystems: the TinyNPU hardware itself, the quantizer and model-preparation path, the compiler, and the runtime. This organization matches the real engineering boundaries of the stack and makes the movement of tensors, instructions, and packed data explicit.

\subsection{System Overview}
At a high level, the flow is straightforward. A model starts either as a quantized PyTorch module or as a compiler-ready artifact produced by the quantization path. The compiler traces the model, partitions it into accelerator-supported regions and explicit host operations, lowers legal regions into TinyNPU instruction and buffer images, and packages the result as a compiled artifact. The runtime then executes that artifact step by step, loading the accelerator when needed, invoking host-side operations when needed, and checking expected tensors when verification is enabled. Inside the hardware, the unified buffer, streamers, systolic array, and post-processing unit implement the actual matrix-style execution path.

That end-to-end view is important because the main design problem is not only to build a datapath. It is to maintain a clean contract between logical tensors, physical packed layouts, host-side fallback behavior, and the actual hardware pipeline. The rest of this section therefore explains the system in the same order: hardware first, then model preparation, then compilation, then runtime execution and verification.

\subsection{NPU}
The TinyNPU hardware is organized around a control path and a compute path. The control path is responsible for instruction sequencing, host service, and control-state progression. In the current runtime path, most UB/IM staging uses the shared-SRAM host window, while MMIO remains the launch/status control mechanism and compatibility fallback. The compute path consists of the unified buffer, streamer logic, a parameterized $N \times N$ output-stationary systolic array, and the post-processing unit. During active computation, data is read from the unified buffer, oriented by the streamers, consumed by the processing elements, and then drained through the post-processing unit before being written back in packed form.

\subsubsection{Unified Buffer and Physical Roles}

The unified buffer is not just a generic scratchpad in this design. Architecturally, it behaves as a true wide dual-port memory with a masked write path. One read side serves the activation stream, while the other serves the weight stream; occasional control-unit random reads are arbitrated onto the activation-side port. This form was chosen to keep both array feed directions active at the same time. A narrower or less concurrent memory interface would turn buffer access into the bottleneck and leave the systolic array underutilized. Because the buffer is wide and row-oriented, this organization is also compatible with a banked memory implementation rather than requiring a single monolithic custom memory block.

Within that buffer, the names A, B, C, and BIAS refer to physical roles in the datapath. A is the left-fed operand: its tiles are packed so that each unified-buffer row can launch one horizontal activation vector into the array boundary. B is the top-fed operand: its tiles are packed for the vertical weight stream. C is the writeback region: it stores quantized output tiles in the packed format chosen by the output precision, so its physical layout may differ from the logical matrix view. BIAS is a separate region consumed only by the PPU. These names therefore describe how data is stored and consumed by the hardware pipeline, not whether a tensor was logically an activation, a weight, or an intermediate result in the original model. Internal outputs are written back in the physical layout required by their next consumer, including an A-compatible transposed/repacked form for chained matmuls.

\subsubsection{Systolic Array and Skewers}

The array is output-stationary: packed activation words enter from the left edge and move one PE to the right per cycle, packed weight words enter from the top edge and move one PE downward per cycle, and each PE keeps its output accumulator stationary for the full reduction along the $K$ dimension. Only after the last $K$ contribution has been consumed does the array switch into drain mode and shift those accumulated results downward into the post-processing unit. This choice fits the dominant tiled-GEMM workload well because it keeps the reduction local to the PE and avoids repeatedly moving partial sums through the buffer system during the $K$ loop.

The streamer logic between the unified buffer and the array provides the timing skew that makes the systolic wavefront legal. On the activation side, row \(i\) is delayed before it enters the left edge; on the weight side, column \(j\) is delayed before it enters the top edge. The values themselves do not change. What changes is when they enter the array. That timing offset creates the diagonal wavefront expected by the schedule, so the activation and weight pieces that belong together meet at the correct PE in the correct cycle. A tile therefore executes in three phases: feed the packed $K$-steps into the array, allow the wavefront to fill across the remaining diagonals, and then drain the stationary accumulators.

\subsubsection{Processing Element}

The processing element is where mixed precision becomes concrete. Each PE latches one 16-bit packed activation word and the lower 16 bits of the vertical bus as the current packed weight word. In INT16 mode that word holds one signed multiply; in INT8 mode it holds two signed byte multiplies whose products are summed; in INT4 mode it holds four signed nibble multiplies whose products are summed. The resulting 32-bit partial sum is sign-extended and accumulated into a 64-bit stationary register.

One subtle but important implementation detail is that the vertical PE-to-PE path is 64 bits wide even though the compute datapath consumes only 16-bit packed weights. This is not wasted width. During compute, only \texttt{data\_from\_top[15:0]} is interpreted as the incoming packed weight word and the full 64-bit bus is simply delayed and passed downward. During drain, the same physical column bus is re-used to shift full 64-bit accumulators from one row to the next. The shared bus keeps the PE interface uniform across compute and drain instead of requiring a second dedicated drain network.

\subsubsection{Post-Processing Unit}

The post-processing unit (PPU) is best understood as an \(N\)-lane row processor backed by an internal \(N \times N\) tile buffer. During drain, it receives one \(N\)-element accumulator row vector per cycle from the bottom of the systolic array, processes all \(N\) lanes of that row in parallel, and stores the quantized result into its internal tile buffer. Only after the full tile has been captured does it begin writing buffered rows back to the unified buffer. The current PPU therefore is not a second streaming mesh; it is a row-wise post-processing stage followed by buffered writeback.

The PPU loads one output-column bias vector before drain begins, in two chunks of four 32-bit values. That fetch happens while the tail of the current wavefront is still finishing inside the array, so the bias load is largely hidden under the transition from feed/wait into drain rather than paid as a separate row-by-row cost. For each drained row it then evaluates the same requantization pipeline lane by lane:
\[
q = \mathrm{clip}_{p}\left(\mathrm{act}\left(\left((acc + bias) \times multiplier + 2^{shift-1}\right) \gg shift\right)\right)
\]
where \(acc\) is the 64-bit drained accumulator, \(bias\) is a sign-extended 32-bit bias term, \(multiplier\) is the 16-bit fixed-point requantization factor, \(\gg\) is an arithmetic right shift, \(\mathrm{act}(\cdot)\) selects the configured post-processing nonlinearity, and \(\mathrm{clip}_{p}(\cdot)\) saturates to the signed range of the selected output precision. In the current stack, the activation stage supports plain ReLU, an integer sigmoid path derived from a DI-Exp-style integer nonlinear primitive in the spirit of I-LLM \cite{hu2024illm}, and a hardware-friendly hard-GELU approximation used when GELU is lowered into the accelerator \cite{hendrycks2016gelu,kim2021ibert}. Concretely, the datapath performs bias addition, fixed-point multiplication, round-to-nearest when \(shift > 0\), the selected activation, and final saturation to INT16, INT8, or INT4.

Writeback is also precision-dependent. After the tile buffer has been filled, the PPU emits one buffered row per writeback cycle. For INT16 output, each 16-bit lane is written back directly. For INT8 and INT4 output, the PPU shifts the quantized value into the byte or nibble slot selected by the packed write offset, and the unified buffer applies a bit mask so that only that subword is updated while previously written sibling tiles in the same physical word are preserved. The control unit derives this packed write offset from the output tile index: two logical M-tiles share one physical word for INT8, and four logical M-tiles share one physical word for INT4. The writeback path also supports layout selection for chained tensors. When the compiler marks an internal output as the next layer's A-side input, the PPU reformats its buffered tile during writeback into an A-compatible packed layout. For INT16 this behaves like a small tile transpose; for INT8 and INT4 it becomes a transpose-plus-subword repack that places the byte or nibble groups where the left-fed streamer expects them. This layout-aware writeback is part of the validated hardware contract.

\subsubsection{Control and Instruction Format}

This behavior is programmed through a fixed-width 256-bit instruction format in instruction memory. The current \texttt{MATMUL} instruction encodes the opcode, base addresses for the A, B, C, and bias regions in the unified buffer, the tile counts along $M$, $K$, and $N$, and the PPU configuration fields for multiplier, shift, activation, input precision, output precision, hard-GELU metadata, and output-layout selection. The control unit therefore does not fetch a stream of tiny scalar operations; it fetches a coarse-grained instruction that describes one tiled matrix-multiplication region and then steps through clear, feed, wait, drain, bias-load, and writeback states internally. This ISA was kept deliberately simple because the supported workload is highly predictable. A finer-grained instruction stream would add fetch and decode overhead without providing much useful flexibility in the current compiler contract.

\subsection{Quantizer and Model Preparation}
The quantization path exists to prepare trained models into coherent compiler-ready artifacts. The goal here is not to solve quantized training or bit allocation as standalone research problems. The goal is narrower and more practical: produce models whose weights, activations, ranges, signedness, and optional mixed-precision choices are explicit enough that the compiler can lower them without losing numerical coherence.

The role of QAT is clearer in the activation study of Section~\ref{sec:activation-study}. In the \texttt{W16A16} example there, the reference MNIST classifier reached 71.54\% in FP32, post-training quantization reduced that to 35.53\%, and QAT recovered it to 69.98\%, which is close to the FP32 reference. In that example, QAT is not merely validating the export path. It is the stage that adapts the trained model to the quantized deployment contract that the final artifact must obey.

This part of the stack also includes the mixed-precision helper. Its purpose is limited and should be stated clearly: it is not a global bit-allocation solver. Instead, it provides a practical one-layer-at-a-time sensitivity workflow that can generate usable mixed-precision configurations, recalibrate them, and pass the result onward to the compiler-ready conversion path. In other words, it is a way to prepare plausible deployable models for this hardware rather than a claim of optimal bit allocation.

The QAT path is more structured than simply toggling fake quantizers. It has three stages. First, a trained FP32 checkpoint is calibrated to estimate per-layer activation ranges and to initialize activation and weight scales. Second, the model is fine-tuned with quantized layers in place, so activations and weights see fake-quantized values during training. Third, the trained QAT model is frozen into a compiler-ready form with explicit integer weights, scale factors, output precisions, and fused multiplier/shift metadata. In TinyNPU, QAT is therefore the stage that turns a trained floating-point model into a deployable quantized artifact rather than the stage that performs lowering itself.

This is also where accelerator-specific activation approximations are prepared before lowering. For the integer sigmoid and hard-GELU experiments, QAT trains against the same broad activation contract that the compiler later freezes into the deployed artifact. In the GELU case, this includes learning a per-layer hard-GELU input-domain shift that is emitted as instruction metadata for the PPU. The broader direction is motivated by work such as I-LLM and I-BERT, even though the final TinyNPU hard-GELU form is a project-specific hardware approximation rather than a verbatim paper operator \cite{hu2024illm,kim2021ibert}.

The important role of this subsystem is therefore not to solve quantization research as a whole. It is to provide a disciplined bridge from trained or quantized models into forms that the TinyNPU compiler can lower without losing track of numerical meaning. The resulting artifacts are then carried through segmented compilation and RTL-backed execution, with compiled-host and RTL behavior checked against each other on sampled inputs rather than stopping at an offline export boundary.

\subsection{Compiler}
The compiler is responsible for converting model structure into a mixed execution structure. Rather than generating a monolithic accelerator-only program for the whole model, the compiler constructs an \texttt{ExecutionPlan} whose steps are typed as \texttt{NpuSegment}, \texttt{HostOp}, or \texttt{VerifyTensor}. \texttt{NpuSegment} groups accelerator-supported matrix-style work. \texttt{HostOp} keeps unsupported or intentionally host-side work separate. \texttt{VerifyTensor} allows compiler-owned expected tensors to be checked later at runtime.

This representation is the main reason the stack remains analyzable. Unsupported behavior is not folded into a fake accelerator-only view, and accelerator boundaries are not implicit. That is a deliberate policy: if someone wants to target TinyNPU effectively, the model and compiler flow should reflect the actual hardware boundaries rather than pretending full-model coverage already exists.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.92\textwidth,height=0.72\textheight,keepaspectratio]{diagrams/compiler_stack_graphviz.pdf}
    \caption{Compiler and runtime flow used by the TinyNPU JIT path.}
    \label{fig:compiler-stack}
\end{figure}

Figure~\ref{fig:compiler-stack} summarizes the implemented software flow. A model enters either directly as a supported quantized PyTorch module or through a compiler-ready conversion path built around \texttt{tinynpu\_quant}. The JIT frontend traces the model with FX, partitions it into a mixed execution plan, validates host operations, and lowers each accelerator segment through the memory planner and \texttt{SegmentCompiler}. The resulting \texttt{CompiledArtifact} contains the execution plan, per-segment IM/UB binaries, expected tensors for verification, and the memory-layout information required by the runtime. The current emitter/runtime boundary keeps this explicit by generating static program tables in C and executing them through one shared runtime implementation.

Packing is part of the compiler contract as well. The backend uses role-specific physical layouts for activation-side tensors, weight-side tensors, output buffers, and bias payloads. These are physical storage forms rather than semantic tensor meanings, but they are essential because legal execution depends on matching memory layout to the real feed directions and writeback format of the NPU. A useful correction came late in the project: internal NPU outputs can no longer be treated as if ordinary C-style writeback were automatically consumable by a following A-side input. The compiler now annotates the required physical output layout explicitly, and the emitted instruction stream carries that selection into the RTL writeback path.

The important compiler policy is not the cycle-by-cycle tile schedule but where accelerator regions begin and end. The partitioner emits a mixed execution plan and forces a segment boundary whenever execution must return to the host, a verification point is requested, or an operation is outside the current NPU contract. As a result, host-side work such as dequantization, softmax, mean, layout restoration, and verification is represented directly as \texttt{HostOp} or \texttt{VerifyTensor} steps rather than being buried inside an accelerator launch.

Memory planning makes those execution boundaries explicit in the buffer layout. The unified buffer is divided into a static zone and a dynamic zone: weights and biases receive globally assigned packed addresses that can be shared across segments, while activations and outputs are placed in a per-segment dynamic region with liveness-based reuse. The planner also treats layout as a real physical requirement. If a tensor must be emitted in an A-compatible layout for an internal chained consumer, that requirement is carried in the memory report rather than being inferred informally. At host-visible boundaries, the flow preserves a host-friendly INT16 representation where possible, especially for boundaries that will be read back or consumed by host operations. The reason is simple: when data returns to the host, the runtime prefers to hand off the highest precision the datapath can produce instead of requantizing early and making host-side functions such as softmax or verification unnecessarily lossy.

\subsection{Runtime and Verification}
The runtime executes the compiled artifact as a mixed host-accelerator loop. For each step in the execution plan, it either launches an accelerator segment, executes an explicit host operation, or checks a verification tensor. The key point is that host-side work is never hidden. If a step remains on the host, it remains visible in the plan and in the benchmark accounting.

The runtime also owns repeated-run behavior. It can preserve reusable state such as static unified-buffer contents and preloaded instruction-memory images across runs. This does not remove all setup work, but it makes repeated inference more realistic and exposes a measurable optimization path for reducing avoidable overhead.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.9\textwidth,height=0.58\textheight,keepaspectratio]{diagrams/runtime_state_graphviz.pdf}
    \caption{Host-side runtime flow for executing a compiled TinyNPU artifact.}
    \label{fig:runtime-state}
\end{figure}

Figure~\ref{fig:runtime-state} shows the same loop from the host-side orchestration perspective. For accelerator segments, the runtime first packs logical tensors into the physical layouts expected by the datapath, stages vectors into the unified buffer (shared-SRAM path by default, MMIO fallback available), launches execution, waits for completion, and then reads and unpacks the resulting vectors back into logical tensors. Host-side operations such as requantization, dequantization, softmax, mean, and shape movement appear explicitly in the same loop rather than being hidden behind the accelerator interface. With the newer output-layout support, the host-side runtime no longer has to repair certain internal layout mismatches itself: when the compiler marks a chained tensor as needing A-format storage, the hardware segment writes it back in that form directly.

Verification is built into this same path. Compiler-owned expected tensors can be checked during execution, which means correctness is not limited to end-of-run output comparison. Benchmarking is also integrated here: the runtime can record hard RTL cycle counts for accelerator run windows while separately accounting for software-side overhead such as packing, MMIO traffic, and readback.

\subsection{Current Scope Boundaries}
The current system targets a narrow but concrete contract. The mixed-precision helper produces compiler-ready configurations rather than solving the global bit-allocation problem. The compiler supports a bounded operator set and routes remaining work through host steps. The evaluation focuses on one end-to-end model and one controlled kernel benchmark, which is enough to study the interaction between architecture, compilation, and runtime without overstating workload coverage.

\section{Experimental Setup and Design} 
Evaluation uses three evidence sources. Unit and regression tests verify correctness incrementally across the stack, including compiler transformations, execution-plan behavior, memory planning, and host-side operator semantics. RTL-backed cocotb tests then exercise the hardware/software path directly and expose the control-unit performance counters during execution. A reproducible Yosys flow provides rough logic and DSP estimates for the current top-level design, with and without abstracted memories.

Host emulation and RTL simulation execute the same compiled plan, so the same artifact can be checked both for tensor-level correctness and for hardware-visible cost. The control unit reports both total run-window cycles and per-state cycle and entry counts, which makes it possible to separate time spent in feed, wait, drain, writeback, and related control states.

The evaluation platform is the CV32E40P example testbench wired to TinyNPU through two host interfaces. First, MMIO control registers are exposed at \texttt{0x3000\_0000} for command/status traffic (including \texttt{CMD\_RUN}). Second, a shared SRAM window is exposed for data movement: UB is mapped at \texttt{0x3100\_0000} and IM at \texttt{0x3200\_0000}, each as \(8192\) words \(\times\) \(16\) bytes. The testbench decode maps those CPU addresses to TinyNPU \texttt{host\_shared\_*} signals and converts IM accesses to internal IM word space starting at \texttt{0x8000}. In the current runtime configuration, UB/IM staging uses this shared window by default, while MMIO memory commands remain available as an explicit fallback path.

The empirical results are organized around two benchmark views. The first is a controlled multi-tile GEMM chosen to force tiling on the current array and expose the split between array throughput and integration overhead. The second is a model-level case study built from two compiler-ready artifacts that expose how execution-plan shape changes delivered speedup.

The reported metrics keep those cost categories separate. Accelerator cycles are taken from the RTL run window of the NPU segment itself. Packing, MMIO traffic, tensor movement, and host-only operators are accounted for outside that run window. The CPU baselines used later in the report are analytic reference models for the same logical work, not measurements from a physical comparison platform.

The runtime-coverage measurements use one fixed execution setup so the rows are directly comparable. Runtime validation uses \texttt{autoverify} against compiler-owned expected tensors when explicit verify steps are not emitted.

\subsection{Activation Study}
\label{sec:activation-study}

To evaluate activation behavior, we ran a small MNIST classifier with three hidden convolution layers, mean pooling, and a final fully connected layer, then replaced the hidden GELU path with TinyNPU's hardware-friendly hard-GELU. The outcome is clear. In \texttt{W8A8}, hard-GELU is too lossy: the FP32 reference reached 71.54\%, the matching \texttt{W8A8} PTQ model fell to 12.14\%, and QAT recovered only to 54.44\%. Keeping the hard-GELU boundary at INT16 changed the picture: PTQ improved to 35.53\% and QAT recovered to 69.98\%, which is close to the FP32 reference. The practical conclusion is that hard-GELU needs higher post-activation precision to remain accurate.

In the full-memory run, Yosys also infers 8192 \texttt{RAM128X1D} and 944 \texttt{RAM256X1S} primitives. These figures are still FPGA-style synthesis proxies rather than final silicon area, but they are sufficient to place the design in a non-trivial hardware range.

\section{Conclusion and Future Work}
TinyNPU is a working mixed-precision accelerator platform with a usable hardware architecture, a PyTorch-oriented model-preparation flow, a segmented compiler/runtime, explicit host-boundary handling, repeated-inference support, verification infrastructure, and benchmark tooling that separates compute from overhead.

The results show a clear split between compute capability and integration cost. The systolic core delivers strong throughput on tiled GEMM, but practical execution still pays a noticeable cost for segmentation, state movement, packing, and host-visible boundaries. The mixed-precision path is practical for preparing compiler-ready models, and the current evaluation focuses on one end-to-end model and one controlled kernel benchmark.

The immediate priorities are to bring more models onto the stack, reduce avoidable runtime and interface overhead, increase the size of useful accelerator regions, and use measured workloads to justify changes in unified-buffer and instruction-memory sizing. The hard-GELU study sharpened that agenda: richer nonlinearities are feasible in the current PPU, but only if the precision-memory tradeoff is handled explicitly. In the evaluated MNIST GELU study, W8A8 hard-GELU was too lossy, while an INT16-preserving variant recovered nearly all of the FP32 accuracy but exceeded the present unified-buffer budget. That makes selective precision retention around sensitive activation boundaries a more useful next target than simply adding more nonlinear modes. On the hardware side, synthesis and timing characterization should now be extended beyond rough FPGA proxies toward a more complete implementation study. On the software side, compiler work should continue to target the boundary cases and data-movement costs that show up in measured workloads.

\section{Implemented Activation Algorithms}
\label{app:activation-algorithms}

This appendix records the activation functions as they were actually implemented in the TinyNPU software golden path and mirrored in the RTL primitives. These algorithms therefore describe the deployed integer contract rather than an idealized floating-point reference.

\begin{algorithm}[H]
\caption{TinyNPU DI-Exp}
\begin{algorithmic}[1]
\Require Integer input $x_{in}$, integer parameters $m_I > 0$, $k_I \ge 0$
\Ensure Integer approximation of an exponential-like decay
\State $m_F \gets m_I + (m_I \gg 1) - (m_I \gg 4)$
\State $s_I \gets \left(2^{k_I} + \lfloor m_F / 2 \rfloor \right) \mathbin{/} m_F$
\State $t \gets -s_I$
\State $q \gets \lfloor x_{in} / t \rfloor$
\State $r \gets x_{in} - q \cdot t$
\State $u \gets (r \gg 1) - t$
\State $y \gets u \gg q$
\State \Return $\max(0, y)$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[H]
\caption{TinyNPU DI-Sigmoid}
\begin{algorithmic}[1]
\Require Integer input $x_{in}$, integer parameters $m_I$, $k_I$, output precision parameter $p_{out}$, optional smoothing factor $\alpha_{smooth}$
\Ensure Integer sigmoid approximation in a signed $p_{out}$-bit probability domain
\State $x_s \gets \lfloor x_{in} / \alpha_{smooth} \rfloor$
\State $e_0 \gets \textsc{DI-Exp}(0, m_I, k_I)$
\If{$x_s \ge 0$}
    \State $e \gets \textsc{DI-Exp}(-x_s, m_I, k_I)$
    \State $n \gets e_0$
\Else
    \State $e \gets \textsc{DI-Exp}(x_s, m_I, k_I)$
    \State $n \gets e$
\EndIf
\State $d \gets e_0 + e$
\State $q_{max} \gets 2^{p_{out}-1} - 1$
\State \Return $\left(n \cdot q_{max} + \lfloor d/2 \rfloor\right) \mathbin{/} d$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[H]
\caption{TinyNPU Hard-GELU}
\begin{algorithmic}[1]
\Require Signed integer input $x_{in}$, activation-domain shift $k_x$, slope parameters $(n_s, k_s)$ approximating $1.702 \approx n_s / 2^{k_s}$
\Ensure Signed integer hard-GELU result in the same activation domain
\State $S \gets 2^{k_x}$
\State $t_3 \gets 3S$
\State $t_6 \gets 6S$
\State $s \gets \textsc{RoundShiftSigned}(x_{in} \cdot n_s, k_s)$
\State $g \gets \min(\max(s + t_3, 0), t_6)$
\State $y \gets \textsc{RoundDivSigned}(x_{in} \cdot g, t_6)$
\State \Return $y$
\end{algorithmic}
\end{algorithm}

\section{Benchmark Cost Model}
\label{app:benchmark-model}
The benchmark path used in this project combines measured RTL cycle counts with an analytic reference-CPU model. The goal is not to claim a definitive comparison against a specific commercial processor. Instead, it is to make the accelerator results interpretable by separating four different questions:
\begin{enumerate}
    \item How much logical work would a scalar CPU have performed for the same operation?
    \item How many cycles did the accelerator actually spend in its measured run window?
    \item How much extra cost was introduced by packing, MMIO traffic, and host-side orchestration?
    \item How much work still remained on the host because the compiled plan retained helper operations outside accelerator segments?
\end{enumerate}

To support that split, benchmark entries are grouped into four buckets:
\begin{enumerate}
    % cpu_replaced
    \item \textbf{CPU-replaced work \((C_{\text{cpu}})\):} logical work that the CPU would have performed instead of an accelerator segment. For matrix multiplication, this includes scalar reads, multiply-accumulate work, loop-control branches, optional bias addition, requantization, and output writes.
    % npu_compute
    \item \textbf{Accelerator compute \((C_{\text{npu}})\):} measured RTL cycle counts collected during the accelerator run window with performance monitoring enabled.
    % npu_overhead
    \item \textbf{Accelerator overhead \((C_{\text{ovh}})\):} software-visible and protocol-visible work required to use the accelerator, such as packing, interface writes, interface reads, instruction loading, and readback.
    % host_remaining
    \item \textbf{Remaining host work \((C_{\text{host}})\):} all helper work outside accelerator segments that still executes on the host. This includes \texttt{quantize}, \texttt{dequantize}, \texttt{requantize}, \texttt{softmax}, \texttt{mean}, \texttt{reshape}, \texttt{transpose}, \texttt{layout\_restore}, \texttt{im2col}, and alias-style movement helpers.
\end{enumerate}

Primitive work is first counted in an abstract instruction-class form. The counted fields include reads, writes, adds, multiplies, branches, shifts, clamps, bitwise operations, divides, nonlinear helper calls, interface writes, and interface reads.

An analytic CPU model is then applied to those counts:
\[
\texttt{cycles} = \sum_i \texttt{count}_i \times \texttt{cost}_i
\]

The three reference models currently used in the project are shown in Table~\ref{tab:benchmark-cost-models}. These are implemented directly in the benchmark code and reuse the same counted trace; only the CPI assumptions change.

Using those symbols, the main reported speedups are:
\begin{align*}
S_{\text{pure}} &= \frac{C_{\text{cpu}}}{C_{\text{npu}}} \\
S_{\text{full}} &= \frac{C_{\text{cpu,full}}}{C_{\text{npu}} + C_{\text{ovh}} + C_{\text{host}}}
\end{align*}

Here \(C_{\text{cpu,full}}\) is the modeled CPU-only cost of the logical program. It starts from \(C_{\text{cpu}}\) and then adds only host-side functions that would still belong to the logical CPU program, such as \texttt{softmax}, \texttt{mean}, \texttt{reshape}, \texttt{transpose}, or an unfused host nonlinearity. Accelerator-serving helpers such as \texttt{quantize}, \texttt{dequantize}, \texttt{requantize}, \texttt{im2col}, \texttt{layout\_restore}, and alias-style movement helpers are excluded from that numerator.

This is why \(S_{\text{full}}\) is the more useful end-to-end comparison in the current report. \(S_{\text{pure}}\) isolates the raw compute advantage of the array itself, while \(S_{\text{full}}\) compares the delivered accelerator execution against a CPU-only execution of the same logical model.
